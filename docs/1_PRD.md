# CloudGuessr PRD (총정리본)

## 0. 문서 목적

* 2개월 내(홍보주간 부스) 시연 가능한 **클라우드 지오게서형 미니게임**을 안정적으로 구현하기 위한 제품 요구사항 정의서.
* 우선순위는 “연구적 완성도”보다 **현장 운영 안정성 / 직관성 / 처리량**에 둔다.

---

## 1. 한 줄 요약

* 캠퍼스 포인트클라우드 전체 지도에서, 주어진 부분 클라우드(문제지)가 나온 위치를 맞추면 시스템이 정합(ICP)을 수행해 점수를 매기는 체험형 게임.

---

## 2. 배경 및 맥락

* 개발/시연 환경

  * OS: Ubuntu 22.04, amd64
  * Middleware/런타임: ROS 2
  * 패키지: **ament_cmake**(이미 빈 패키지 생성 완료)
* 데이터

  * 캠퍼스 주요 도로(약 650m x 650m) 주행으로 구축한 포인트클라우드 맵(.ply)
* 사용 장소

  * 대학교 홍보주간 동아리 부스(단일 노트북, 1인 플레이 + 관중 다수)

---

## 3. 사용자 및 시나리오

### 사용자(플레이어)

* 포인트클라우드 경험이 없는 일반 방문자
* 1회 플레이 시간 30초~2분 선호

### 운영자(스태프)

* 라운드 시작/리셋/설명 진행
* 빠른 회전율(처리량)과 문제 발생 시 즉시 복구 가능한 구조 필요

---

## 4. 목표 / 비목표

### 목표

* 1~2분 내 규칙 이해 후 즉시 플레이 가능
* “맵/부분 관측/정합” 개념을 체감할 수 있는 피드백 제공
* 2~3시간 연속 운영에서 크래시/프리즈 없이 반복 가능
* 점수판(선택)으로 관중 몰입 유도

### 비목표

* SLAM/지도 품질 평가/연구용 고정밀 정합
* 난이도 높은 6DoF 수동 정합(일반인 기준 과부하)
* 외부 네트워크 의존(현장 환경 변동성 최소화)

---

## 5. 게임 모드 결정

### 기본 모드: Mode A(원클릭 위치 선택)

* 플레이어 입력: **전체 맵에서 한 점 클릭(위치만)**
* 시스템 처리: 클릭 위치 주변 ROI를 crop → 다중 yaw 가설로 정합 → 최고 점수 채택

#### 다중 yaw 가설(백엔드)

* 클릭 위치는 초기 translation으로 사용
* yaw는 사용자가 입력하지 않고 시스템이 후보를 생성

  * 후보 예: 8방향(45° 간격) 또는 16방향(22.5° 간격)
* 각 yaw 후보에 대해 ICP 실행 후 결과 점수 최대값을 사용

> 결정 근거: RViz와 Open3D를 병행하는 구조에서, 사용자가 RViz에서 yaw까지 지정하는 것은 비직관적이며 실수/혼선을 유발할 가능성이 큼.

---

## 6. 핵심 게임 루프(UX)

1. 라운드 시작: 문제지(query) 표시
2. 플레이어는 문제지를 관찰(Open3D)
3. 플레이어는 전체 맵에서 “여기다” 싶은 곳을 클릭(RViz)
4. 시스템이 채점(ROI crop + 다중 yaw ICP)
5. 점수/오차/정답 위치를 보여주고 다음 라운드

### 화면 구성(초기 MVP)

* Open3D 창: query 클라우드만 표시(회전/확대/이동)
* RViz 창: 전체 맵 표시 + 클릭 입력
* 간단 HUD(터미널/간단 GUI): 라운드 번호, 점수, 안내 문구, 리셋/다음

---

## 7. 기능 요구사항(Functional Requirements)

### FR-1 라운드 관리

* 라운드 시작/다음/리셋 기능
* 라운드별 query 파일 및 메타데이터 로드
* 타임아웃(선택): 일정 시간 입력 없으면 자동 안내 또는 다음

### FR-2 지도 표시 및 입력

* 맵(PointCloud2)을 RViz에 안정적으로 렌더링
* RViz에서 클릭 좌표를 수신

  * 기본 토픽: `/clicked_point` (geometry_msgs/PointStamped)

### FR-3 채점(정합)

* 클릭 위치 중심으로 ROI crop 수행
* query 및 ROI에 대해 다운샘플/노이즈 완화 전처리
* 다중 yaw 후보로 ICP 수행 후 최고 점수 선택
* 수렴 실패/비정상 결과는 실패로 판정하고 낮은 점수 처리

### FR-4 결과 피드백

* 점수(0~5000)
* 거리 오차(정답 위치 vs 클릭 위치)
* (선택) RViz에 정답 위치/입력 위치 마커 표시

### FR-5 운영 편의

* 단축키 또는 버튼으로 다음 라운드/리셋
* 로그 저장(라운드, 입력, 점수, 처리 시간, 성공/실패)

---

## 8. 점수(Scoring) 정의

### 점수 범위

* 만점: **5000점**

### 구성 요소(권장)

* 정합 품질: ICP 결과의 fitness(inlier ratio) + rmse
* 시간 페널티(선택): 채점 시간 또는 입력 제한시간 초과
* 실패 판정: 수렴 실패, inlier ratio 임계 미달, rmse 임계 초과 → 매우 낮은 점수

### 점수 함수(정의 수준)

* score = 5000 × Q(fitness, rmse) − Penalty(time)
* Q는 0~1로 정규화된 품질 함수
* 임계값/정규화 방식은 구현 단계에서 튜닝하되, PRD 기준은 다음을 만족해야 한다.

  * “정답 근처”는 눈에 띄게 높은 점수
  * “엉뚱한 곳”은 대부분 낮은 점수
  * 동일 라운드에서 운에 의한 점수 분산이 과도하지 않을 것

---

## 9. 데이터(라운드) 규격

### 라운드 생성 방식(결정)

* 전체 맵(.ply)에서 CloudCompare 등으로 **crop 생성**

### 라운드 폴더 구조(권장)

* `round_{i}/`

  * `query.ply` : query 클라우드(로컬 프레임)
  * `round.yaml` : 메타데이터

### round.yaml 필드(권장)

* `gt_pose_in_map`: {x, y, z, yaw(optional)}
* `difficulty`: easy | medium | hard
* `roi_radius`: (권장) 기본 ROI 반경(m)
* `notes`: (선택) 힌트/설명용

> GT pose 저장 목적: 거리 오차/결과 표시/ICP 실패 시에도 피드백 유지.

## 10. 시스템 설계(ROS 2)

### 노드 구성(MVP)

1. `cloudguessr_map_server` (rclpy)

* map.ply 로드
* 다운샘플(voxel)
* PointCloud2로 변환하여 publish

2. `cloudguessr_round_manager` (rclpy)

* 라운드 상태머신
* query 로드 및 Open3D 뷰어 구동(또는 별도 프로세스)
* `/clicked_point` 구독
* ROI crop → ICP 실행 → 점수 산출
* 결과 publish(점수/마커/로그)

3. `cloudguessr_backend` (Python 모듈)

* 전처리(다운샘플/필터)
* ROI crop(공간 인덱싱: voxel/octree 등)
* ICP + 다중 yaw 스윕
* 점수 계산 및 실패 판정

### 주요 토픽(권장)

* 입력: `/clicked_point` (PointStamped)
* 맵: `/cloudguessr/map` (PointCloud2)
* 점수: `/cloudguessr/score` (예: std_msgs/Int32 또는 커스텀)
* 결과 마커: `/cloudguessr/markers` (MarkerArray)

## 11. 비기능 요구사항(NFR)

* 안정성

  * 2~3시간 부스 운영 중 크래시/프리즈 0회 목표
  * 예외 발생 시 라운드 매니저가 자동 복구(라운드 리셋 가능)

* 성능

  * 채점 결과 표시까지 목표: **2초 이내**(권장)
  * 핵심 전략: “전체 맵 정합 금지” + “ROI crop + 다운샘플 + 캐시”

* 운영성

  * 다음 라운드/리셋이 1~2번 조작으로 가능
  * 로그 파일로 사후 분석/개선 가능

* 재현성

  * 같은 입력이면 점수가 큰 폭으로 흔들리지 않도록 랜덤 최소화

---

## 12. 리스크 관리(우선순위 포함)

### R1. 성능 저하(최우선)

* 원인: 맵 규모, ICP 반복, 다운샘플 부족
* 완화: ROI crop 강제, voxel downsample, 캐시, yaw 후보 수 조절

### R2. ICP 실패/오판

* 원인: 초기값이 멀거나 반복 구조(대칭), 밀도 차이
* 완화: 실패 판정 기준 도입, 다중 yaw, ROI 반경 조절, query 난이도 조절

### R3. UI가 게임처럼 보이지 않음

* 원인: RViz 자체가 개발자 도구 느낌
* 완화: HUD(점수판/라운드 안내) 강화, 결과 마커 시각화, 운영 멘트/설명 스크립트

### R4. 현장 운영 리스크(오클릭/리셋 누락)

* 완화: 큰 버튼/단축키, 타임아웃, 데모 모드(자동 채점)

### R5. 좌표계/스케일 혼선

* 완화: map frame 정의 고정, 전처리 파이프라인 스크립트화, round.yaml로 GT 관리

## 13. 성공 지표

* 평균 1라운드 플레이(설명+선택+채점) 2분 내

* 현장 장애 0회 또는 즉시 복구 가능

## 14. 마일스톤(2개월)

* W1: 데이터 전처리 파이프라인 + 오프라인 채점 스크립트(ROI crop + ICP + 점수)
* W2: ROS2 노드화(map publish, clicked_point 수신) + RViz 표시
* W3: 라운드 매니저(상태머신) + 결과 출력(점수/거리오차)
* W4: 라운드 세트 제작(난이도 라벨링) + 기본 튜닝(ROI/voxel/yaw 후보)
* W5: HUD/운영 버튼/단축키 + 마커 시각화
* W6: 스트레스 테스트(연속 100라운드) + 성능 최적화
* W7: 부스 운영 스크립트/동선 정리 + 플레이테스트
* W8: 기능 동결 + 백업 플랜(데모 모드/안전 설정)
